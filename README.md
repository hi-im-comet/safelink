# CoolLink AI

**폭염 취약가구와 일반 시민을 위한 저전력 기후안전 에이전트** — 프론트엔드 MVP.

폭염 위험을 개인의 주거환경·생활패턴·냉방 접근성·돌봄 공백과 결합해 판단하고,
일반 사용자에게는 *오늘 어떻게 안전하고 효율적으로 버틸지*, 관리자에게는 *오늘 누구를 먼저 도와야 할지*를 안내합니다.

> 위험도 계산은 **규칙 기반 코드**가 담당하고, LLM은 전화 스크립트·방문 체크리스트·행동계획·보고서 같은 **문서 생성**에만 사용합니다. LLM 호출을 줄이는 것이 곧 저전력·탄소중립 전략입니다.

## 실행 방법

```bash
npm install
npm run dev
```

접속: [http://localhost:3101](http://localhost:3101/)

기본 개발 포트는 `3101`입니다. `3100`은 다른 로컬 프로젝트와 충돌할 수 있어 이 프로젝트는 `package.json`의 `dev/start` 스크립트에서 `3101`을 사용합니다.

빌드/프로덕션:

```bash
npm run build && npm start
```

> 백엔드 없이 **mock data + 클라이언트 상태(localStorage)** 로 전체 데모가 동작합니다.
> Node 18.18+ / Next.js 15 / React 19 / TypeScript / Tailwind CSS.

## 화면

| 경로 | 설명 |
|------|------|
| `/` | 랜딩 — 큰 기후 비주얼 + 관리자 / 사용자 앱 / 서비스 소개 진입 3개 |
| `/about` | 서비스 소개 — 하는 일, 매일 아침 갱신, AI 사용 구조(코드/비전/LLM), 저전력, 확장 가능성 (담백한 카드형) |
| `/admin` | **관리자 대시보드(데스크톱)** — KPI 카드 클릭 → 필터/날씨 상세, 우선순위 리스트(긴급 최상단·출처/사유/대상 태그), 필터, **오늘 대응 요약 생성**(모달) |
| `/admin/households/[id]` | 가구 상세 — 왜 지금 우선 대응인지, 위험 요인·추천 조치, 위험도 구성(35/25/25/15), 조치 기록, 전화 스크립트·방문 체크리스트 생성·복사 (`H-001` = 94점) |
| `/user` | **사용자 앱(모바일)** — 기본/보호자 **두 모드**. 매일 오전 6시 갱신형 홈(날짜·지역·날씨·위험도·시간대 행동계획) |
| `/user/profile` | 주거환경 설정(고정 프로필·접기/펼치기) — 기본주거·냉방차광·창·**생활패턴·더위 민감도·냉방 성향**·취약도 + **자연어 생활 메모(AI 보조 분석)** + 창 사진 분석. 모드별 내 집/부모님 집 |
| `/user/help` | 도움 요청 — **연락 대상 선택**(119/지자체/보호자/생활지원사) + 사유 2단계 선택 → 출처·대상·사유 태그로 관리자에 전달 |
| `/user/mypage` | 마이페이지 — **보호자 기능 사용 여부 토글** |

> 사용자 앱은 두 모드입니다. **기본 모드** = 내 집 기준 폭염 행동계획. **보호자 모드**(마이페이지에서 켬) = 부모님·가족 집 기준 안부 확인. 끄면 가족 관련 카드는 보이지 않습니다.
> 주요 화면 우하단에 **"AI 절감 92.1%"** 플로팅 버튼이 있으며, 클릭하면 전체 분석·규칙 기반 처리·LLM 호출·캐시 재사용·LLM 미사용 처리율·평균 출력 토큰을 확인할 수 있습니다.

## 발표 데모 흐름 (약 3분)

1. 랜딩(`/`)
2. 사용자 앱(`/user`)
3. 주거설정(`/user/profile`)
4. 도움요청(`/user/help`)
5. 관리자 대시보드(`/admin`)
6. 가구 상세(`/admin/households/H-001`)
7. AI 절감 토글
8. 서비스 소개(`/about`)

> 데모를 처음 상태로 되돌리려면 관리자 상단의 **데모 초기화** 버튼을 누르세요. (localStorage 초기화)

## 위험도 규칙 (코드 계산)

```
최종 위험도 = 날씨 35% + 개인 취약도 25% + 주거·냉방 25% + 돌봄 공백·도움요청 15%
등급: 80↑ 긴급 확인 · 60~79 전화·방문 · 40~59 행동요령 · 40↓ 모니터링
도움 요청 가구는 점수와 별개로 우선순위 최상단에 고정됩니다.
```

규칙 엔진은 `lib/risk/score.ts` 에 있으며 LLM을 호출하지 않습니다.

## 구조

```
app/                  라우트 (랜딩 / admin / user)
components/           UI (히어로 레이어·우선순위 행·위험도 구성·문서 생성·모바일 다이얼 …)
lib/
  types.ts            공통 타입
  risk/score.ts       규칙 기반 위험도 엔진 + 우선순위 정렬
  mock/               가구·날씨·사용자·지표 mock data
  llm/templates.ts    문서 생성기 (LLM이 담당하는 유일한 영역)
  store/AppState.tsx  공유 상태 (도움 요청·조치 기록·LLM 호출, localStorage)
  api/index.ts        데이터 접근 레이어 — 지금은 mock, 본문만 fetch로 교체하면 백엔드 연결
```

**API 연결 지점:** 화면은 `lib/api`(읽기·문서 생성) 와 `lib/store/AppState`(상태 변경)만 호출합니다.
백엔드가 준비되면 `lib/api/index.ts` 의 함수 본문을 `fetch(...)` 로 바꾸면 컴포넌트는 그대로 둡니다.

## 디자인

연두·초록·민트가 dominant. 위험 신호만 흙빛 warm(ember/amber)으로 처리해 등급이 멀리서도 또렷이 구분됩니다.
히어로/배경은 사진 없이 CSS·SVG·gradient로 온도 레이어를 구성했고, 카드는 정보 성격에 따라 크기·형태를 다르게 했습니다.
토큰은 `tailwind.config.ts` 와 `app/globals.css` 에 정리되어 있습니다.

> MVP는 폭염만 구현합니다. 한파·홍수·미세먼지·산불연기는 위험도 구조를 그대로 두고 재난 모듈만 교체하는 방식으로 **향후 확장**합니다.
